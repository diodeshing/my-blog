# 如何用 AI 写代码：不是替代你，是加速你

AI 写代码的能力越来越强，但大多数人的用法是错的。

## 常见误区

**❌ 把 AI 当搜索引擎用**
> "帮我写一个完整的电商平台"
> → 生成 2000 行代码，你根本看不懂，也不敢用

**❌ 完全不审核就复制粘贴**
> → 代码能跑，但有安全隐患、性能问题、风格不一致

**❌ 用它做它不擅长的事**
> → "帮我判断这个 UI 设计好不好看" — AI 没有审美

## 正确用法

### 1. 让 AI 写样板代码

```
你：写一个 Express.js 的用户 CRUD 接口，用 MongoDB
AI：（生成 router 文件，包含增删改查）
你：（检查 + 修改 + 集成到项目）
```

省时间：40 分钟 → 5 分钟

### 2. 让 AI 解释你不理解的代码

```
你：解释这段代码在做什么（贴代码）
AI：逐行解释 + 标注关键点
```

### 3. 让 AI 写测试

```
你：为这个函数写单元测试（贴函数）
AI：生成测试用例，包含边界条件
```

### 4. 让 AI 帮你调试

```
你：这段代码报了 XXX 错误（贴代码 + 错误信息）
AI：分析原因 + 给出修复方案
```

## 提示词技巧

### 好的提示词

```markdown
用 Python 写一个函数，要求：
- 输入：文件路径列表
- 功能：并行读取所有文件内容，返回字典 {文件名: 内容}
- 使用 asyncio + aiofiles
- 包含错误处理
```

**要点：** 语言、输入输出、具体技术、边界条件

### 差的提示词

```
帮我写个读文件的程序
```

→ AI 不知道用什么语言、什么方式、要处理什么边界情况。

## 实际工作流

```
1. 你设计架构（AI 不擅长这个）
2. 你拆分任务（每个函数做什么）
3. AI 生成每个函数的代码
4. 你审查、修改、整合
5. AI 写测试
6. 你运行测试，修复问题
```

**AI 负责手，你负责脑。**

## 工具推荐

| 工具 | 特点 | 适合场景 |
|------|------|---------|
| GitHub Copilot | IDE 内联补全 | 写代码时实时建议 |
| Claude | 长上下文、理解力强 | 解释复杂代码、重构 |
| ChatGPT | 通用、生态广 | 快速原型、学习新概念 |
| Cursor | IDE 集成 AI | 全流程 AI 辅助编程 |

## 总结

> AI 不会让不会写代码的人变成程序员，但**会让会写代码的人效率翻倍**。关键是你要做架构师，让 AI 做执行者。
